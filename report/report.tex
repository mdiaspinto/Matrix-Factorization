\section{Description of the Adopted Solution}
% ============================================================

\subsection{Problem Setting}

With explicit feedback, a user tells you directly what they think: a four-star rating on a movie is unambiguous. Implicit feedback is messier. A user watches a movie, and all we observe is that the interaction happened. If a user \emph{didn't} watch something, we don't know whether they disliked it or simply never heard of it. The model has to learn from this kind of incomplete information.

Hu et al.~\cite{hu2008} proposed a way to handle this. We cite their paper here because the eALS algorithm we were assigned to implement (He et al.~\cite{he2016}) is a direct optimization of their formulation. The loss function, the confidence weighting, and the ALS solver all come from Hu et al., making it central to our project.

Their idea is to treat every user-item pair as a data point, but weigh them differently. Observed interactions get high confidence ($c_{ui} = 1 + \alpha$), while missing entries get a small uniform weight $c_0$. The model minimizes:

\begin{equation}
\mathcal{L} = \sum_{u}\sum_{i} c_{ui}\left(r_{ui} - \mathbf{p}_u^\top \mathbf{q}_i\right)^2 + \lambda\left(\|\mathbf{P}\|^2 + \|\mathbf{Q}\|^2\right)
\label{eq:loss}
\end{equation}

where $r_{ui} = 1$ for observed interactions and $0$ otherwise, $\mathbf{P} \in \mathbb{R}^{M \times k}$ and $\mathbf{Q} \in \mathbb{R}^{N \times k}$ are the user and item latent factor matrices, and $\lambda$ controls regularization.

The trouble is that the sum in Equation~\ref{eq:loss} runs over all $M \times N$ pairs, including the vast number of zeros. Standard ALS handles this by alternating: fix $\mathbf{Q}$, solve for each user $\mathbf{p}_u$, then fix $\mathbf{P}$ and solve for each item $\mathbf{q}_i$. But each user update requires building and inverting a $k \times k$ matrix that depends on all $N$ items. The per-user cost is $O(Nk^2)$, which gets expensive quickly as the item catalog grows.

\subsection{The eALS Algorithm}

He et al.~\cite{he2016} proposed element-wise ALS (eALS), which avoids iterating over all items by restructuring the update. Rather than solving a $k$-dimensional linear system per user, eALS updates one latent factor at a time. The key idea is to precompute:

\begin{equation}
\mathbf{S}^q = \mathbf{Q}^\top \mathbf{Q} \in \mathbb{R}^{k \times k}
\end{equation}

This is a small matrix that summarizes the aggregate contribution of all items. Once you have $\mathbf{S}^q$, the closed-form update for user $u$, factor $f$ is:

\begin{equation}
p_{uf} = \frac{\displaystyle\sum_{i \in \mathcal{I}_u} (c_{ui} - c_0)\, q_{if}\, \hat{r}_{ui}^{(f)} \;-\; c_0\left(\mathbf{p}_u^\top \mathbf{S}^q_{:,f} - p_{uf}\, S^q_{ff}\right)}{\displaystyle\sum_{i \in \mathcal{I}_u} (c_{ui} - c_0)\, q_{if}^2 \;+\; c_0\, S^q_{ff} \;+\; \lambda}
\label{eq:update}
\end{equation}

where $\hat{r}_{ui}^{(f)} = 1 - \sum_{f' \neq f} p_{uf'}\, q_{if'}$ is the residual with factor $f$ removed, and $\mathcal{I}_u$ is the set of items user $u$ interacted with. The first term in the numerator sums only over observed items ($|\mathcal{I}_u|$ terms). The second term uses $\mathbf{S}^q$ to account for all items without touching them individually. This brings the per-user cost down to $O(|\mathcal{I}_u| \cdot k + k^2)$.

\subsection{Why This Approach}

eALS has two properties that make it well suited for this project. First, the element-wise update sidesteps the $O(Nk^2)$ cost of standard ALS, which matters once the item catalog gets large. Second, the algorithm parallelizes naturally: when updating user factors, each user's computation only reads from $\mathbf{Q}$ and $\mathbf{S}^q$, with no dependencies between users. So you can distribute users across Spark workers with no algorithmic changes, and the same applies when updating items.

We implemented the algorithm in Python using NumPy (single-machine reference) and PySpark RDDs (distributed), and evaluated on MovieLens datasets using the leave-one-out protocol from the paper.

\section{Algorithm Design}
% ============================================================

\subsection{Overall Structure}

Each iteration has two symmetric half-steps:

\begin{enumerate}
\item \textbf{User update.} Compute $\mathbf{S}^q = \mathbf{Q}^\top \mathbf{Q}$. For each user $u$, update $p_{u0}, p_{u1}, \ldots, p_{u,k-1}$ one at a time using Equation~\ref{eq:update}. After updating $p_{uf}$, refresh a prediction cache so that $\hat{r}_{ui}^{(f+1)}$ can be computed cheaply.

\item \textbf{Item update.} Compute $\mathbf{S}^p = \mathbf{P}^\top \mathbf{P}$. For each item $i$, update $q_{i0}, q_{i1}, \ldots, q_{i,k-1}$ using the symmetric formula (swap $\mathbf{P}$ and $\mathbf{Q}$).
\end{enumerate}

The prediction cache is a practical optimization. Without it, after updating $p_{uf}$ you would need to recompute $\mathbf{p}_u^\top \mathbf{q}_i$ for all of user $u$'s items before updating the next factor. With the cache, you just patch it: $\text{cache}[i] \mathrel{+}= \delta \cdot q_{if}$, where $\delta$ is the change in $p_{uf}$. This costs $O(|\mathcal{I}_u|)$ instead of $O(|\mathcal{I}_u| \cdot k)$.

\subsection{Complexity}

Per iteration, the cost breaks down as:
\begin{itemize}
\item Precomputing $\mathbf{S}^q$: $O(Nk^2)$
\item Updating all users: $O\!\left(\sum_u |\mathcal{I}_u| \cdot k + M k^2\right) = O(|\mathcal{R}|\,k + Mk^2)$
\item Precomputing $\mathbf{S}^p$ and updating all items: $O(|\mathcal{R}|\,k + Nk^2)$
\end{itemize}
Total: $O(|\mathcal{R}|\,k + (M+N)k^2)$, where $|\mathcal{R}|$ is the number of observed interactions. Standard ALS costs $O((M+N)Nk^2)$. For MovieLens 100K, $|\mathcal{R}| \approx 100\text{K}$ and $MN \approx 1.6\text{M}$, so eALS is already roughly $16\times$ cheaper per iteration.

\subsection{Spark Parallelization Strategy}

During the user-update half-step, no user's update depends on any other user's. Each one only reads $\mathbf{Q}$ and $\mathbf{S}^q$, both of which are fixed for the whole half-step. We take advantage of this in Spark:

\begin{enumerate}
\item The driver computes $\mathbf{S}^q$ ($k \times k$, cheap) and broadcasts both $\mathbf{Q}$ and $\mathbf{S}^q$ to all workers.
\item An RDD of \texttt{(user\_id, item\_indices)} is partitioned across workers. Each worker processes its chunk of users independently.
\item Updated user vectors are collected back to the driver.
\end{enumerate}

The item-update step works the same way: broadcast $\mathbf{P}$ and $\mathbf{S}^p$, partition items, collect results.

The only point where workers have to wait for each other is between half-steps: all user updates must finish before item updates start, because item updates need the new $\mathbf{P}$.

\subsubsection{Data Loading and Preprocessing}

The \texttt{data\_loader.py} module takes raw MovieLens files and produces sparse matrices ready for training.

\begin{lstlisting}[caption={Binarization and minimum-interaction filtering.}]
def binarize(interactions, threshold=0.0):
    binarized = [
        (u, i, 1.0, ts)
        for u, i, r, ts in interactions
        if r > threshold
    ]
    return binarized

def filter_min_interactions(interactions, min_user=5, min_item=0):
    prev_count = len(interactions) + 1
    while len(interactions) < prev_count:
        prev_count = len(interactions)
        if min_user > 0:
            user_counts = defaultdict(int)
            for u, i, v, ts in interactions:
                user_counts[u] += 1
            interactions = [
                (u, i, v, ts) for u, i, v, ts in interactions
                if user_counts[u] >= min_user
            ]
        # symmetric for items
    return interactions
\end{lstlisting}

All ratings become binary (line 3): any observed interaction becomes 1. The filtering loop (lines 9--18) runs iteratively because removing items can push a user below the threshold and vice versa. In practice the loop converges in one or two passes.

For the train/test split, we use leave-one-out: each user's most recent interaction (by timestamp) goes to the test set, and the rest is training data. This follows the protocol in~\cite{he2016}.

\subsubsection{Element-wise User Update (NumPy)}

\begin{lstlisting}[caption={Core element-wise update for user factors.}]
def _update_users(P, Q, R_csr, Sq, c_obs, c0, reg, M, k):
    for u in range(M):
        item_indices = R_csr[u].indices
        if len(item_indices) == 0:
            continue
        Q_u = Q[item_indices]            # |I_u| x k
        pred_cache = Q_u @ P[u]          # |I_u| predictions

        for f in range(k):
            q_f = Q_u[:, f]
            hat_r = 1.0 - pred_cache + P[u, f] * q_f

            numer = (c_obs - c0) * np.dot(q_f, hat_r)
            numer -= c0 * (P[u] @ Sq[:, f] - P[u, f] * Sq[f, f])

            denom = (c_obs - c0) * np.dot(q_f, q_f) + c0 * Sq[f, f] + reg

            old_val = P[u, f]
            P[u, f] = numer / denom
            pred_cache += (P[u, f] - old_val) * q_f
\end{lstlisting}

Line 7 computes $\mathbf{p}_u^\top \mathbf{q}_i$ for all observed items in one matrix-vector product. The inner loop (line 9) iterates over factors. Line 11 computes $\hat{r}_{ui}^{(f)}$ by removing factor $f$'s contribution from the cached prediction. Lines 13--14 implement the numerator from Equation~\ref{eq:update}: the first term sums over observed items, the second uses $\mathbf{S}^q$ for the all-items contribution. Line 20 patches the cache in $O(|\mathcal{I}_u|)$ rather than recomputing everything.

\subsubsection{Spark RDD Parallelization}

\begin{lstlisting}[caption={User update distributed via Spark RDD.}]

Q_bc = sc.broadcast(Q)
Sq_bc = sc.broadcast(Sq)

        Q_u = q_local[item_indices]
        pred_cache = Q_u @ p_u
        for f in range(k):

updated_users = user_rdd.mapPartitions(update_user_partition).collect()

    P[u] = p_u
\end{lstlisting}

The code mirrors the NumPy version. The difference is that the outer loop over users (line 10) runs inside a Spark partition, and partitions are processed in parallel on different cores or machines. Lines 2--4 broadcast $\mathbf{Q}$, $\mathbf{S}^q$, and $\mathbf{P}$ once to all workers, rather than shipping them per task. Line 20 collects the results back to the driver.

We use \texttt{mapPartitions} rather than \texttt{map} so that each worker deserializes the broadcast variables once per partition rather than once per user.

\subsubsection{Loss Computation}

Computing the loss naively means iterating over all $M \times N$ pairs. We avoid this with a decomposition:

\begin{lstlisting}[caption={Efficient loss computation using the trace trick.}]
def _compute_loss(P, Q, R_csr, c_obs, c0, reg, M):
    Sp = P.T @ P
    Sq = Q.T @ Q
    # All-pairs contribution assuming r=0, weight=c0
    loss = c0 * np.trace(Sp @ Sq)
    # Correct for observed entries
    for u in range(M):
        items = R_csr[u].indices
        preds = Q[items] @ P[u]
        loss -= c0 * np.dot(preds, preds)
        residuals = 1.0 - preds
        loss += c_obs * np.dot(residuals, residuals)
    loss += reg * (np.sum(P**2) + np.sum(Q**2))
    return loss
\end{lstlisting}

The idea: first, pretend every entry has $r_{ui}=0$ and weight $c_0$. Line 5 computes this in $O(k^3)$ via $c_0 \cdot \text{tr}(\mathbf{S}^p \mathbf{S}^q)$, which equals $c_0 \sum_u \sum_i (\mathbf{p}_u^\top \mathbf{q}_i)^2$. Then lines 9--12 fix the observed entries: subtract the wrong $c_0 \cdot \text{pred}^2$ and add the correct $c_\text{obs} \cdot (1-\text{pred})^2$. The loop only touches observed entries, so the total cost is $O(k^3 + |\mathcal{R}|\,k)$ instead of $O(MNk)$.

\subsubsection{Evaluation}

\begin{lstlisting}[caption={Leave-one-out evaluation with Hit Rate and NDCG.}]
def evaluate(P, Q, train_matrix, test_dict, top_k=10):
    R_csr = train_matrix.tocsr()
    hits = 0
    ndcg_sum = 0.0
    for u, test_item in test_dict.items():
        scores = Q @ P[u]
        scores[R_csr[u].indices] = -np.inf  # mask training items
        top_k_items = np.argpartition(scores, -top_k)[-top_k:]
        top_k_items = top_k_items[np.argsort(-scores[top_k_items])]
        if test_item in top_k_items:
            hits += 1
            rank = np.where(top_k_items == test_item)[0][0] + 1
            ndcg_sum += 1.0 / np.log2(rank + 1)
    return {"HR": hits / len(test_dict),
            "NDCG": ndcg_sum / len(test_dict)}
\end{lstlisting}

For each test user, we score all items (line 6), set training items to $-\infty$ so they can't appear in the ranking (line 7), and find the top-$K$ with \texttt{argpartition} (line 8), which is $O(N)$ rather than $O(N \log N)$. If the held-out item appears in the top-$K$, we record a hit and compute the NDCG contribution from its rank position (line 13).

\section{Experimental Analysis}
% ============================================================

We used three MovieLens datasets: 100K (943 users, 1{,}682 items), 1M (6{,}040 users, 3{,}706 items), and 10M (69{,}878 users, 10{,}677 items), all with binary implicit feedback. Users with fewer than 5 interactions were removed. Unless noted otherwise, we set $\alpha = 1$, $c_0 = 1$, $\lambda = 0.01$. All experiments ran on a MacBook with an Apple Silicon CPU and 8 cores; Spark ran in local mode using all cores.

\subsection{Convergence}

We trained eALS with $k=32$ for 50 iterations, measuring both training loss and recommendation quality at regular intervals.

The loss (Figure~\ref{fig:convergence}, left) drops steeply in the first 5 iterations, losing about 20\% of its initial value, then flattens. \section{Discussion: Strengths and Weaknesses}
% ============================================================

\subsection{Strengths of eALS}

\textbf{Speed per iteration.} The $\mathbf{S}^q$ precomputation removes the need to loop over missing entries. On MovieLens 100K, each iteration takes under a second. Standard ALS, which builds a $k \times k$ system per user involving all $N$ items, would be roughly an order of magnitude slower.

\textbf{Easy parallelization.} User updates don't depend on each other (given fixed $\mathbf{Q}$ and $\mathbf{S}^q$), so distributing them across Spark partitions required no changes to the algorithm itself. The same update code runs on each worker; only the data shipping differs. Porting from NumPy to Spark was mechanical.

\textbf{Fast convergence.} The algorithm gets close to its best recommendation quality within 5--10 iterations (Section 3.1). Combined with the low cost per iteration, total training time stays reasonable even without a cluster for medium-sized datasets.

\textbf{Low memory per worker.} To update one user, a worker needs the user's own $k$-vector, the $|\mathcal{I}_u| \times k$ submatrix of $\mathbf{Q}$, and the $k \times k$ matrix $\mathbf{S}^q$. For $k=32$ and a user with 100 interactions, that comes to about 25 KB.

\subsection{Weaknesses and Limitations}

\textbf{Driver bottleneck.} The full $\mathbf{P}$ and $\mathbf{Q}$ matrices live on the driver and get broadcast every iteration. For ML-100K this is negligible (about 400 KB for $\mathbf{Q}$ at $k=32$), but a dataset with 10 million items would mean broadcasting around 2.5 GB per half-step. At that point, communication would dominate computation.

\textbf{Sequential factor updates.} The element-wise update processes factors $f=0,1,\ldots,k-1$ one after another, because each one modifies the prediction cache that the next one reads. This inner loop cannot be parallelized. For large $k$, this becomes the tightest bottleneck.

\textbf{Uniform weighting of missing data.} We use a single weight $c_0$ for all missing entries. The original paper also proposes item-popularity-based weighting, where items that more users interacted with get a higher missing-data weight. This can improve quality, but we did not implement it due to time constraints.

\textbf{Python loop bottleneck at 10M scale.} We could not train on the full MovieLens 10M (70K users, 10M interactions) in reasonable time. The element-wise update runs as a Python \texttt{for} loop over users and factors, and at 25\% of 10M users (17K users, 2.5M interactions), each iteration already took 9.6 seconds. Extrapolating, the full dataset would need roughly 40 seconds per iteration and over 30 minutes for 50 iterations. A Cython or C extension for the inner loop would fix this on one machine; on a real cluster, the Spark version would handle it.

\textbf{No online update evaluation.} The paper's title mentions ``online recommendation,'' and eALS does support incremental updates: when new interactions arrive, you can update only the affected user/item vectors instead of retraining from scratch. We only implemented batch training and did not test this.

\subsection{Comparison with MLlib}

MLlib ALS slightly outperformed our eALS on all metrics (Table~\ref{tab:baseline}). Two things probably explain this. First, MLlib is heavily optimized at the JVM level with native BLAS, while our updates run in Python loops. Second, MLlib uses standard block-wise ALS, which solves the full $k$-dimensional system per user and may converge to a slightly better solution for the same iteration count. The gap is small enough (3\% relative) that it does not change the practical picture, especially given eALS's lower per-iteration cost.
